Security & Identity

The user acts, not the agent

Identity propagates end-to-end via OAuth on-behalf-of so downstream systems authorize the human, with policy enforced at every hop.

1
User signs in to portal
Okta / Entra OIDC — MFA enforced, session controls applied, user identity established
2
Portal calls super-agent
Bearer token (user's access token) — identity is the human, not a service account
3
Agent calls MCP server
OAuth token exchange (RFC 8693) → scoped downstream token, user identity preserved
4
MCP server calls SaaS
Downstream token identifies the user, not a service account — SaaS RBAC/FLS is enforced
5
SaaS enforces its own RBAC
User only sees / acts on data they're entitled to — no privilege escalation possible
Action Authorization (OPA)
Every tool call passes through Open Policy Agent — "may this user run this tool on this record?" Evaluated before execution.
PII Redaction
Sensitive data is redacted before prompts leave the VPC. Secrets stored in Vault / KMS — never injected into prompts.
Prompt-Injection Defense
Tool outputs treated as untrusted data. System prompts are signed. Allow-listed tool surface reduces attack area.
Immutable Audit Trail (7yr)
Every agent decision, tool call, and user confirmation stored immutably for 7 years. Full replay for compliance reviews.
Human-in-the-Loop
Actions above risk threshold require explicit user confirmation before execution. Read-by-default; writes require approval.
